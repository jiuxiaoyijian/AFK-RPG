# 项目进展记录 v1.2

## 1. 本轮定位

本轮承接 `v1.1`，开始正式推进“系统边界拆分”。

这次没有继续堆新玩法，而是优先做第一阶段重构：

- 把资源、研究树、收益修正从 `GameManager` 中抽离
- 保留兼容层，避免一次性重构过深导致原型失稳
- 继续维护版本化项目进展记录

## 2. 本轮目标

目标：

- 降低 `GameManager` 的职责密度
- 为后续继续拆 `LootSystem / CombatSystem / SaveManager` 打基础
- 在不改变现有玩法体验的前提下，提升结构清晰度

## 3. 本轮完成内容

### 3.1 新增 `MetaProgressionSystem`

本轮新增了独立 autoload：

- `scripts/autoload/meta_progression_system.gd`

它现在负责：

- 资源状态：`gold / scrap / core / legend_shard`
- 研究状态：`research_levels`
- 资源收益修正
- 分解回收修正
- 研究升级校验与执行
- 研究树筛选、详情与概览文本
- 研究对战斗属性的增益汇总
- 元成长相关存档数据打包与恢复

这意味着“资源与研究成长”已经开始脱离 `GameManager`，形成更明确的系统边界。

### 3.2 `GameManager` 降级为兼容层

本轮没有粗暴删除旧接口，而是先把 `GameManager` 中对应方法改成代理：

- `grant_rewards()`
- `get_meta_progression_bonuses()`
- `get_adjusted_reward_count()`
- `get_adjusted_salvage_scrap()`
- `get_research_*()`
- `can_upgrade_research()`
- `upgrade_research()`

同时：

- 装备自动分解 / 手动分解改为通过 `MetaProgressionSystem` 发放铁屑
- 角色总战斗属性中的研究加成改由 `MetaProgressionSystem` 注入

这样做的意义是：

- 先迁移实现，再逐步迁移 API
- 当前原型功能不需要因为重构而返工

### 3.3 关键调用点已迁移

本轮已把几个核心调用点改接到 `MetaProgressionSystem`：

- `SaveManager`
- `HUD`
- `ResearchPanel`
- `CombatSystem` 首通奖励
- `LootSystem` 材料掉落
- `OfflineSystem` 离线材料结算

这说明新的边界不只是“创建了个新脚本”，而是已经被实际使用。

### 3.4 存档边界更清晰

本轮后，`SaveManager` 不再自己手写资源与研究字段的存取细节，而是改为：

- 调 `MetaProgressionSystem.build_save_data()`
- 调 `MetaProgressionSystem.load_save_data(payload)`

这使得存档职责更清晰：

- `SaveManager` 负责总流程
- `MetaProgressionSystem` 负责自己的状态序列化

## 7. 当前项目状态

截至 `v1.2`，项目已经从“单一大管家式 `GameManager`”往更标准的边界迈出了一步：

- `GameManager` 更偏世界流程、背包与装备
- `MetaProgressionSystem` 更偏资源、研究与收益修正
- `SaveManager` 更像总调度，而不是把所有字段都亲自管理

这属于一次结构性进展，而不是纯内容堆叠。

## 8. 当前已知问题

- `GameManager` 中仍保留部分兼容代理方法，还没有彻底瘦身
- `LootSystem` 与 `CombatSystem` 目前仍偏原型化，尚未拆出更细的职责对象
- 章节视觉表现和图鉴追踪仍未推进

## 9. 推荐下一步

按当前节奏，下一轮建议优先做：

- 继续第二阶段边界拆分，把掉落结算或节点推进再拆一层
- 或者补 `图鉴 / 目标追踪 / 定向 farming 提示`
- 或者开始做章节视觉差异化，让当前两章的结构升级更可见
