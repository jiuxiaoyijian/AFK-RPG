# 项目进展记录 v1.7

## 1. 本轮定位

本轮承接 `v1.6` 的期望值可视化，继续把图鉴推荐从“静态估算”推进到“估算 + 真实样本”并存的状态：

- 记录玩家真实刷图掉落样本
- 在图鉴中展示实战统计
- 继续维护版本化项目进展记录

## 2. 本轮目标

目标：

- 让图鉴不只显示理论期望值，还能看到真实掉落结果
- 为后续做“统计面板 / 最近掉落回顾 / 更真实推荐修正”打基础
- 保持当前系统边界尽量稳定，不额外引入新复杂系统

## 3. 本轮完成内容

### 3.1 图鉴系统新增真实掉落样本记录

本轮在 `LootCodexSystem` 中加入了节点掉落样本记录。

当前每次节点结算时，系统都会记录：

- 该节点累计刷了多少次
- 掉了多少件装备
- 出现过哪些底材
- 出现过哪些词条
- 出现过哪些传奇特效

这意味着图鉴系统现在已经不只是“收集和推荐”，还开始沉淀玩家自己的实战数据。

### 3.2 LootSystem 已接入统计记录

`LootSystem.process_node_loot()` 在完成本次节点掉落结算后，会把本次实际掉出的装备列表交给 `LootCodexSystem.record_node_loot()`。

因此：

- 就算这次没有掉装备，节点结算次数也会被记录
- 如果掉了装备，则会进一步记录对应的底材 / 词条 / 传奇样本

这让样本数据天然和现有自动战斗循环对齐，不需要额外的手动操作。

### 3.3 图鉴详情新增“实战统计”

现在在 `底材 / 词条 / 传奇` 三类图鉴详情中，除了：

- 推荐刷图
- 推荐原因
- 期望收益

还会新增：

- `实战统计(全局)`
- `实战统计(推荐节点)`

展示内容包括：

- 已刷多少次
- 见到目标多少次
- 实际约多少次才能见到 1 次
- 对推荐节点可额外显示约多少分钟见到 1 次

这样图鉴已经开始具备“理论推荐”和“个人实测”双轨信息。

### 3.4 图鉴摘要新增样本量

图鉴顶部摘要本轮新增了：

- `样本 X 次`

这里的样本数指当前累计记录的节点结算次数。

这样玩家能更快判断：

- 当前实战统计是不是还太少
- 推荐与实测之间的偏差是否已经有足够样本支撑

### 3.5 新统计已纳入存档

本轮新增的节点掉落统计数据已经并入 `LootCodexSystem.build_save_data()` / `load_save_data()`。

这意味着：

- 关闭游戏后不会丢失样本
- 后续可以在此基础上继续扩展最近掉落记录、图表、过滤器等功能

## 4. 本轮关键实现点

逻辑侧重点：

- `LootCodexSystem.record_node_loot()`：记录节点掉落样本
- `LootCodexSystem.get_observed_entry_stats()`：统计指定条目的实战命中情况
- `LootCodexSystem._append_observed_stats_lines()`：把实测数据拼进图鉴详情
- `LootSystem.process_node_loot()`：在结算后把本次掉落交给图鉴系统归档

## 7. 当前项目状态

截至 `v1.7`，图鉴与掉落推荐系统已经具备：

- 分类浏览
- 目标追踪
- build 相关推荐
- 期望值可视化
- 真实掉落样本统计

现在玩家已经能同时看到：

- 系统理论上推荐去哪刷
- 理论上大概要刷多久
- 自己实际刷了多少次、出了几次

这让掉落追求开始从“黑盒”变成“可解释、可观察”的系统。

## 8. 当前已知问题

- 当前实战统计仍是文本展示，还没有单独的统计面板
- 全局统计目前按玩家全部刷图样本汇总，不区分“理论上是否可掉落该目标”
- 还没有记录最近若干次掉落明细，因此无法回看短期波动

## 9. 推荐下一步

按当前节奏，下一轮建议优先做：

- 做独立的“掉落统计面板”，展示最近样本、推荐节点样本和全局样本
- 或给推荐结果加入“理论值 vs 实测值”的偏差提示
- 或继续拆 `GameManager`，把节点推进和背包装备管理分离
