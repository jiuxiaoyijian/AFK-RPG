# 桌面挂机项目总控与 AI 执行说明

## 1. 项目目标

本项目目标是做出一款可在 `Godot 4` 中运行的、适合 `Steam 桌面常驻` 的横版自动战斗挂机游戏原型，并在后续扩展为可玩的正式版本。

设计参考来自《英雄没有闪》的公开可观察机制，但不照搬其手游化结构，改造成更适合 PC 桌面挂机的形式：

- 自动横版推进战斗
- 刷装和词条构筑
- 单核心技能流派
- 插件式辅助系统
- 在线常驻挂机 + 离线补偿
- 低打扰、高反馈、可长期成长

## 2. 目标体验

玩家打开游戏后，不需要高频操作，而是通过以下行为获得乐趣：

- 观察角色自动推进和击杀反馈
- 通过装备掉落不断优化 build
- 根据当前卡点调整主技能、被动、模块与关卡目标
- 在挂机过程中积累资源，并周期性做成长决策
- 在工作或浏览网页时将游戏缩小到桌面角落，保持低打扰运行

## 3. 设计原则

所有后续 AI 在执行时必须遵守以下原则：

1. 不要把项目做成动作游戏。
战斗必须默认自动进行，玩家输入仅用于菜单、构筑和手动挑战可选项。

2. 不要把项目做成传统放置数值点击器。
核心乐趣应来自 `刷装 -> 选词条 -> 调整 build -> 推进更高关卡`。

3. 不要一开始做过大内容量。
先完成可闭环的 MVP，再逐步增加流派、敌人和系统。

4. 所有系统都要服务 build 变化。
如果某个系统无法改变战斗构筑，它的优先级应下降。

5. 尽量配置驱动。
技能、敌人、装备词条、掉落、章节、研究树等应尽量以 `JSON` 或 `Resource` 方式驱动。

2. `02_核心战斗与系统循环设计.md`

## 5. 推荐的 MVP 范围

第一阶段只做一个“可玩原型”，必须包含以下内容：

- 1 个可运行主场景
- 1 条横版自动推进战斗流程
- 3 个核心流派
- 2 个战术技能槽
- 4 个装备部位
- 随机装备掉落与词条
- 章节推进与 Boss
- 自动分解基础规则
- 离线结算
- 基础研究树

第一阶段不要做的内容：

- 多角色切换
- 联机
- 大量剧情演出
- 复杂城镇系统
- 过多 UI 动效
- 20 种以上技能树

## 6. 推荐的内容分期

### M1 原型闭环

目标：做出完整的“挂机战斗 -> 掉落 -> 换装 -> 推图”闭环。

必须完成：

- 自动移动、索敌、攻击、死亡、推进
- 装备掉落和背包
- 自动穿戴或手动换装
- 3 个流派的最小实现
- 普通怪、精英、Boss
- 离线收益

### M2 构筑强化

目标：让 build 有明显差异。

必须完成：

- 被动技能系统
- 战术模块系统
- 更完整的词条池
- 自动分解规则
- 多关卡刷取目标

### M3 长线成长

目标：建立中长期追求。

必须完成：

- 阶段突破
- 研究树
- 高阶装备和传奇词条
- 更高难度循环

## 7. 核心成功标准

如果项目达到以下标准，可认为“原型成功”：

- 玩家 5 分钟内能看懂基本玩法
- 30 分钟内能体验至少 2 次明显 build 改变
- 自动挂机过程中每 1 到 3 分钟至少产生一次有效决策
- 玩家能感受到“当前 build 为什么强/弱”
- 卡关时能通过换词条或调整技能解决，而不是只能等数值

## 8. AI 执行硬性要求

所有 agent 在开发时必须遵守：

- 优先使用 `Godot 4` 原生节点和简单结构
- 优先保证数据结构清晰，再追求特效
- 优先实现功能正确，再补动画和 UI
- 所有配置字段必须与 `05_配置表字段说明.md` 对齐
- 所有系统必须支持后续扩展，不要写死具体技能名

## 10. 推荐开发顺序

严格按下面顺序执行，避免先堆内容再返工：

1. 搭主场景与自动战斗主循环
2. 完成角色、敌人、基础伤害结算
3. 接入章节推进和掉落
4. 接入装备、词条、背包
5. 加入 3 个核心流派
6. 加入离线收益和自动分解
7. 加入研究树和阶段成长
8. 再扩内容量

## 11. 最终交付目标

目标不是只输出代码片段，而是让其他 agent 根据本目录文档，直接在 Godot 中完成一个可玩项目原型。

因此本目录所有文档都按以下原则编写：

- 有明确系统边界
- 有配置表字段
- 有 Godot 模块建议
- 有任务拆分
- 有验收标准

如果后续要继续扩写，只能在不破坏当前文档结构的前提下增加补充文档。
