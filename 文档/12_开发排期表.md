# AFK-RPG 开发排期表

## 1. 文档目的

本表用于把当前项目的“问题判断”和“完善计划”拆成可连续执行的开发排期。

它主要服务三件事：

- 明确接下来每一轮开发优先做什么
- 避免再次把精力分散到统计细修、局部 UI 微调和过早包装
- 让后续每一轮进展记录都能回挂到这张表上，方便判断推进状态

本表基于以下文档整理：

- `01_项目总控与AI执行说明.md`
- `08_产品化缺口与后续优先级清单.md`
- `09_P0留存版开发路线图.md`
- `10_题材与世界观包装草案.md`
- `11_首批美术资源清单与外包需求表.md`
- `13_反桃花源修真命名包装表.md`

## 2. 当前判断

截至当前版本，项目已经具备：

- 自动战斗主闭环
- 掉落、背包、研究树、图鉴、离线收益等系统骨架
- 统一 HUD 与二级页面结构
- 本地运行、Web 导出、GitHub Pages 在线试玩

但当前最关键的产品问题仍然集中在：

1. 玩家重新打开游戏后，不够清楚“现在该做什么”
2. build 虽然存在，但目标感和成型路径不够前台化
3. 阶段成长缺少强事件感
4. 战斗虽然能跑，但挂机观看的高光仍然不足
5. 产品包装、美术资源、音频和新手理解层尚未真正接入游戏内

因此后续排期必须优先围绕：

- 回流目标
- build 目标感
- 阶段事件感
- 观赏性
- 产品化包装

## 3. 排期总原则

- 先做 `P0 留存验证层`，再做 `P1 产品包装层`
- 暂缓继续深挖掉率统计、期望值、UI 微调和工程洁癖式重构
- 每一轮开发结束后，都需要新增一份 `文档/项目进展记录/xx_项目进展记录_*.md`
- 如果本轮完成内容无法直接提升“回流动机 / build 感知 / 挂机观感 / 产品辨识度”之一，则优先级应下降

## 5. 分阶段排期

### 5.1 P0-1 回流目标层

目标：

- 玩家重新打开游戏后，10 秒内知道当前最值得做的事

功能项：

- 主 HUD 增加 `当前目标卡`
- 离线结算增加 `推荐下一步`
- 目标追踪卡补 `完成进度 / 缺口说明`
- 增加 `即将解锁内容预告`
- 增加 `今日建议任务` 或 `当前最接近达成项`

建议输出：

- 一个统一的“当前目标”数据结构
- HUD 目标卡 UI
- 离线报告中的目标建议区

验收标准：

- 不打开二级页面也能知道当前推荐目标
- 离线回来后能看到“回来先做什么”
- 至少能覆盖章节推进、研究升级、目标装备/词条三个方向

依赖关系：

- 依赖现有图鉴追踪、研究树、章节进度数据

### 5.2 P0-2 Build 目标层

目标：

- 让 build 从“系统里存在”变成“玩家明确知道下一步追什么”

功能项：

- 三个核心流派的推荐成长路径
- 新手 build 推荐
- 当前 build 缺口诊断
- 下一件目标装备 / 目标词条提示
- 推荐刷图理由展示

建议输出：

- `流派推荐规则` 配置
- `build 诊断` 逻辑
- 图鉴或 HUD 中的推荐信息块

验收标准：

- 卡关时至少能给出 2 到 3 条明确建议
- 新玩家能知道当前流派该优先补伤害、生存还是功能词条
- 推荐目标与当前实际掉落池、章节和研究状态一致

依赖关系：

- 建议在 P0-1 后进行

### 5.3 P0-3 阶段事件层

目标：

- 把章节推进、成长突破和高价值掉落做成“事件”

功能项：

- Boss 首杀结果卡
- 新章节解锁结果卡
- 研究关键升级提示
- 传奇首次获得提示
- 追踪目标达成提示

建议输出：

- 一套统一结果卡模板
- 事件触发与入队管理逻辑
- 事件展示优先级规则

验收标准：

- 关键节点不再只是文字一闪而过
- 玩家会明确感到“我刚完成了一个阶段”
- 结果卡不影响挂机主循环稳定性

依赖关系：

- 可与 P0-2 并行设计，但建议在目标层后落地

### 5.4 P0-4 观赏性增强层

目标：

- 让战斗首页成为值得偶尔驻足观看的挂机舞台

功能项：

- 核心技命中特效或高光文字反馈
- 暴击、处决、连锁、流血结算反馈
- 精英 / Boss 出场与死亡高光
- 稀有掉落闪光与强化提示
- 流派辨识色和彩色战斗锚点

建议输出：

- 轻量级伪特效方案
- 高光反馈颜色与层级规范
- 稀有掉落表现强化

验收标准：

- 玩家瞟一眼能分出普通战斗与高光瞬间
- 三个核心流派至少具备基础视觉辨识
- 稀有掉落和 Boss 战不再淹没在普通战斗节奏里

依赖关系：

- 可部分依赖 P0-3 的结果卡资产和事件体系

### 5.5 P1-1 题材包装接入层

目标：

- 先把题材包装统一到文档与命名层，再逐步落到游戏内

功能项：

- 正式切换为 `反桃花源修真` 方向
- 新增统一命名包装表
- 章节命名、敌人命名、Boss 命名统一
- 流派包装名接入
- HUD / 图鉴 / 结果卡文案统一到题材基调
- 章节主题差异文案接入

验收标准：

- 玩家能感知这不是泛化的“挂机系统原型”
- 世界观词汇在 UI、章节、敌人和掉落里保持一致
- 文档、美术需求与后续实现口径不再互相冲突

当前进度：

- 已完成 `10_题材与世界观包装草案.md` 的题材切换
- 已完成 `11_首批美术资源清单与外包需求表.md` 的方向同步
- 已新增 `13_反桃花源修真命名包装表.md`
- 已完成章节名、敌人名、流派名和高频 UI 文案的首版接入
- 已完成研究描述、README 和主要中低频文本的首版清理
- 下一步进入剩余零散提示词清理，并逐步转入 `P1-2 首批美术资源替换层`

### 5.6 P1-2 首批美术资源替换层

目标：

- 用最小资源量提升观感和辨识度

功能项：

- 主角基础表现资源
- 三流派图标与高光资源
- 两章背景资源
- 关键敌人与 Boss 识别资源
- 结果卡、稀有边框、基础 UI 图标

验收标准：

- 首页截图质量明显提升
- 章节、敌人、流派和稀有度具备第一层识别度

### 5.7 P1-3 新手理解与引导层

目标：

- 降低系统复杂度带来的理解门槛

功能项：

- 第一局基础引导
- build 选择解释
- 卡关提示
- 关键系统解锁提示
- 目标建议解释文字

验收标准：

- 玩家 5 分钟内看懂玩法基础闭环
- 新玩家知道 `I/U/O/P/G` 这些系统入口分别是做什么的

### 5.8 P1-4 Web/展示优化层

目标：

- 提升公开试玩与对外展示质量

功能项：

- Web 端输入和 UI 回归
- 移动端 / 浏览器兼容性回归
- README 增加截图、GIF、系统结构图
- 首页展示素材整理

验收标准：

- 公开试玩地址在主流浏览器下可用
- 仓库首页能在 1 分钟内说明项目卖点与当前完成度

## 6. 暂缓项

以下内容并非不做，而是当前不应继续抢占主排期：

- 更深的掉率统计与期望值扩展
- 继续细拆系统职责和做纯工程式重构
- 继续局部打磨二级页面排版细节
- 大规模世界观正文扩写
- 完整成就、收藏和挑战模式

## 7. 每轮维护要求

从本表建立开始，后续每一轮工作都应同步做两件事：

1. 更新一份新的进展记录
路径：
`文档/项目进展记录/xx_项目进展记录_*.md`

2. 回看本排期表，至少更新以下信息之一：

- 某阶段从 `待开始` 改为 `进行中`
- 某阶段从 `进行中` 改为 `已完成`
- 某阶段新增风险、阻塞项或依赖变化

## 8. 当前建议执行顺序

严格建议按下面顺序推进：

1. `P0-1 回流目标层`
2. `P0-2 Build 目标层`
3. `P0-3 阶段事件层`
4. `P0-4 观赏性增强层`
5. `P1-1 题材包装接入层`
6. `P1-2 首批美术资源替换层`
7. `P1-3 新手理解与引导层`
8. `P1-4 Web/展示优化层`

## 9. 当前结论

当前项目最缺的不是更多系统，而是：

- 更明确的目标感
- 更强的阶段感
- 更好的挂机观感
- 更清楚的产品辨识度

因此这张排期表的意义，不是把功能继续堆厚，而是帮助项目从“结构完整的原型”推进到“能验证留存和产品方向的版本”。
