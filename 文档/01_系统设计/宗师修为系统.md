# 宗师修为系统 — 开发需求案（§1–§15）

> 面向：Godot 4 + GDScript 实现者（含其他 AI 代理）。  
> 主逻辑脚本约定：`scripts/systems/paragon_system.gd`；状态挂载：`GameManager` autoload（`paragon_state` 等）。  
> 设计定位：角色达到**满级**后，击杀/任务等来源的**经验**转化为**宗师修为经验**；修为等级提升获得**修为点**，在四大面板中分配，提供**缓慢、无硬顶**的成长（类比暗黑 3 巅峰）。

---

## §1 背景与目标

### §1.1 项目背景

- 类型：武侠挂机 RPG，2D 横版。  
- 引擎：Godot 4，脚本语言 GDScript。  
- **宗师修为**：满级后的第二条成长曲线；不替代主等级，而是叠加的长期数值与资源效率来源。

### §1.2 商业与设计目标

1. **满级后仍有清晰进度条**：玩家始终看到「下一级修为还要多久」，避免主等级封顶后的空洞感。  
2. **每级可感知**：每升 1 级固定产出 1 点，强化「级级有收获」。  
3. **轻策略分配**：四大类、每类双属性、部分有 50 点上限，促使玩家做 Build 取舍而非无脑全点一项。  
4. **长线付费/消耗钩子（可选）**：重置分配消耗碎银且费用递增，提供回收货币与试错成本。  
5. **与战斗公式一致**：所有战斗向加成必须进入既有 DPS/承伤公式的**正确乘区或加法池**，禁止重复叠乘错误。

## §2 术语表

| 术语 | 含义 |
|------|------|
| 主等级 | 角色常规等级；达到满级阈值后解锁宗师修为（满级阈值由项目主进度定义，可与剧情章节门闸组合）。 |
| 宗师修为等级 N | 修为独立等级；从 0 或 1 开始需与存档字段统一，本文公式中 **N 表示「当前等级」用于计算升到下一级的需求**（见 §11）。 |
| 修为点 | 每升 1 级修为获得 1 点；未分配点保存在 `unspent_points` / `available_points`。 |
| 碎银 | 游戏内基础货币；用于重置分配等。 |
| 四大面板 | 攻击、防御、技巧、辅助；UI 上以 Tab 或中央切换呈现。 |

## §3 范围与非目标

### §3.1 本期包含

- 修为经验获取、升级、发点、分配、上限校验、确认应用、重置（含费用）、持久化与读档校验。  
- 主面板 UI、分配交互、重置确认弹窗、升级与「圆满」反馈（见 §7）。  
- 与 `GameManager` 汇总属性、战斗、掉落/经验加成链路的对接说明（见 §6、§11）。

- 服务器权威校验（若为纯单机则无需）。  
- 与 PVP 平衡的独立赛季清空（除非产品另有「重入江湖」规则，见 §10）。

## §4 体验分层（L1–L5）

### §4.1 L1：一句话幻想（Fantasy）

「突破瓶颈，迈入宗师之境的修行者。」

### §4.2 L1：情绪主轴（Emotion）

- **主情绪**：成长感——修为条在动、数字在涨、偶尔升级有点可点。  
- **次情绪**：策略感——四点面板、双属性、上限与无上限辅助的取舍。

### §4.3 L1：闭环回报（Payoff）

经验条满 → **修为升级** → **获得 1 点** → 在**攻击**面板投入 → 角色面板 **DPS 可见上升** → 玩家能主观感到「这一级没白打」。

### §4.4 L2：时间尺度

| 尺度 | 内容 |
|------|------|
| 秒 | 击杀、结算条即时给修为经验（或合并到同一飘字通道）。 |
| 分 | 修为分段经验条缓慢增长，允许小数经验在 UI 上四舍五入显示。 |
| 时 | 前约 20 级：期望约 **每 2 小时 1 级**（正常挂机+主动游玩混合，见 §11 调参）。 |
| 日 | 期望约 **每日 3–5 级**（中前期）。 |
| 周 | 攻击/防御/技巧侧若偏科，可出现**某一属性逼近 50 点**。 |
| 月 | 战斗三系共 **6×50=300 点** 理论满额后，新增点数主要流入**辅助**（无上限）。 |

### §4.5 L3：策略空间

典型取向（仅示例，非最优解）：  

1. **全攻击**：极限 DPS，生存与暴击稳定性偏弱。  
2. **均衡**：攻击+技巧并行，曲线平滑。  
3. **辅助优先**：碎银/修为获取加速，中后期转战斗。  

系统应通过**上限**与**无上限辅助**让三种取向都「能玩」，但效率不同。

### §4.6 L4：学习曲线（认知负荷）

| 阶段 | 目标 |
|------|------|
| 约 5 min | 理解四大面板与「每点大概加什么」。 |
| 约 1 h | 完成首次 **5 点**分配并感受到面板变化。 |
| 约 10 h | 体验至少一次**重置**（或看到费用与规则说明）。 |
| 约 100 h | 理解「满 300 战斗点 + 辅助无限」的长线结构（不要求算最优解）。 |

### §4.7 L5：仪式感（Juice）

1. **修为升级**：等级数字**翻滚动画** + 短**光效/震屏（极轻）** + 明确音效；获得 **+1 点** 的强调 UI。  
2. **单属性满 50 点**：该项旁显示**金字「圆满」**标签，直至重置或规则变更。  
3. **重置成功**：简短「经脉重塑」类文案 + 碎银扣除飘字。

## §5 核心规则摘要

1. **解锁**：主等级达到**满级**后解锁宗师修为（解锁前不累计修为经验，或累计但不转化——需二选一并在 §10 固定；推荐：**满级后开始累计**）。  
2. **升级**：修为经验达到阈值后升 1 级；**每级 +1 修为点**（连升则连发点）。  
3. **分配**：点数可投入四大面板共 **8 个属性**；**攻击/防御/技巧**下各属性 **最多 50 点**；**辅助**两项**无上限**。  
4. **战斗三系 300 点满后**：新点只能加在辅助（或仍允许点满项点击但无效——推荐**无效并灰显 + 提示**）。  
5. **重置**：已分配点全部退回为未分配点；消耗碎银，**费用随重置次数递增**（公式见 §11）。  
6. **确认式分配**：单次会话内可多次点 +/- 预览，**按「确认」一次性提交**服务器/本地状态（见 §7）。

## §6 架构与集成

### §6.1 模块边界

- **`ParagonSystem`（`paragon_system.gd`）**：纯函数式或静态方法为主——`sanitize_state`、`gain_experience`、`allocate_point`（或批量 `apply_plan`）、`reset_allocations`、`build_combat_bonuses`、`get_experience_to_next_level` 等；**不**直接操作 UI。  
- **`GameManager`**：持有 `paragon_state` Dictionary；在战斗属性合并、经验结算、存档序列化时调用 `ParagonSystem`。  
- **UI 控制器**：只通过 `GameManager` 暴露的 API 读写；禁止场景树反向扒私有字段。

### §6.2 事件与刷新

- 使用现有 `EventBus.paragon_changed(summary: Dictionary)`（若字段不足则扩展 summary，但保持向后兼容）。  
- HUD、角色面板、伤害飘字等订阅该信号刷新。

### §6.3 依赖数据

- 碎银余额：来自经济系统同一数据源。  
- 主等级：来自进度/角色属性。  
- DPS 预览：读取与角色面板相同的聚合属性（含修为加成后的中间结果）。

### §7.1 宗师修为主面板 — ASCII 线框图

**方案说明（边界补全）**：  
- 顶部为**全局修为等级与经验条**；中央为**四大类切换**（同一时间仅展开一类详情）；底部固定**可用点数**与**重置**入口。  
- 每类展示 **2 条属性**，每条含：**当前已分配 / 上限**（辅助显示「∞」或「—」）、**+/-**、**本地预览值**。

### §7.2 属性分配操作细节

1. **+/- 按钮**：每次点击调整的是**本地预览缓冲**（未写入 `paragon_state`）。  
2. **约束实时校验**：  
   - 单属性不能超过 **50**（辅助除外）。  
   - 所有预览增量之和不能超过 `unspent_points`。  
   - 已达战斗系总 cap 时，辅助外属性 **+** 置灰。  
3. **数值预览**：  
   - 每行显示「本属性贡献」+「面板小计」；右侧汇总「预计 DPS / 预计承伤 / 预计暴击率」等，**与角色属性面板同一套计算函数**（避免两套公式漂移）。  
4. **确认按钮**：  
   - 无变更：禁用。  
   - 有变更且合法：点击后**原子应用**——扣点、写 `allocated`、发 `paragon_changed`、播放成功音效。  
   - 失败（并发修改点数不足）：Toast「修为点不足」并刷新状态。  
5. **切换 Tab**：允许保留各 Tab 的未确认预览；若实现困难，则切换前弹轻提示「未确认分配将丢失」——二选一须在 §12 验收中写死。

### §7.3 重置确认弹窗 — ASCII 线框图

**方案说明（边界补全）**：  
- 必须展示**本次将扣除的碎银**、当前余额是否足够、以及**重置后返还的总已分配点**（用于心理安全感）。  
- 确认前二次强调**费用递增**，避免误触。

```
┌─────────────────────────────────────────────┐
│  重置宗师分配                         [ X ]  │
├─────────────────────────────────────────────┤
│  将退还所有已分配修为点，并重新自由分配。      │
│                                             │
│  本次费用:  12,500  碎银                     │
│  重置次数:  第  3  次（费用将递增）            │
│  当前余额:  58,000  碎银   [ 足够 / 不足 ]    │
├─────────────────────────────────────────────┤
│           [ 取消 ]        [ 确认支付并重置 ]   │
```

## §8 数据结构

### §8.1 存档 JSON 形态（逻辑视图）

与实现字段名允许蛇形/驼峰映射，但语义须一致：

```json
{
  "paragon_level": 25,
  "paragon_exp": 12500,
  "exp_to_next": 3906,
  "allocated_points": {
    "attack": { "strength": 10, "attack_pct": 8 },
    "defense": { "health_pct": 5, "defense_pct": 2 },
    "utility": { "crit_rate": 0, "crit_damage": 0 },
    "misc": { "gold_find": 0, "exp_find": 0 }
  },
  "total_points": 25,
  "unspent_points": 0,
  "reset_count": 0,
  "reset_cost": 5000
}
```

说明：

- `total_points`：历史累计获得的修为点总数（可选，用于统计）；**已分配 + 未分配** 应与此一致或可推导。  
- `exp_to_next`：可由 `paragon_level` 推导；存档可冗余存储便于 UI。  
- `allocated_points` 与 `ParagonSystem` 内 `stat_id` 映射表需在实现层维护**单向转换**，禁止 UI 直接改 JSON 深层键。

### §8.2 与代码 `create_default_state` 对齐

若当前工程使用扁平 `allocated: { stat_id: int }`，迁移规则为：

1. 读档：若存在新 JSON 则展开为 stat_id；若仅旧版则保持兼容。  
2. 写档：统一写一种格式（推荐内部仍用 stat_id，导出界面层用分类）。

## §9 完整操作流（升级 → 得点 → 分配 → 确认）

以下为用户主路径，需可录屏验收：

1. **前提**：角色已达满级，系统解锁宗师修为。  
2. **击杀怪物**：结算将一部分经验划入 `paragon_exp`（或先加通用经验再转化——实现择一，全文以「修为经验池」描述）。  
3. **修为条满**：触发升级流程：  
   - `paragon_level += 1`（或连升循环）；  
   - `unspent_points += gained_levels`；  
   - 播放 §4.7 升级仪式；  
   - 推送 `paragon_changed`。  
4. **打开修为面板**：显示新 `unspent_points`。  
5. **点选「攻击」Tab**，对「力量」「攻击力%」各点若干 **+**。  
6. **右侧预览**显示 DPS 变化；若点超上限或点超可用点，**+** 失效并飘字原因。  
7. **点击「确认分配」**：  
   - 服务端/本地校验通过 → 提交变更 → 扣减 `unspent_points` → 更新 `allocated_points`。  
8. **关闭面板**：战斗中的属性立即生效（下一击伤害变化可感知）。

## §10 边界情况与规则

| 场景 | 期望行为 |
|------|-----------|
| **点数溢出** | 连升多级时，`unspent_points` 一次增加多级；分配确认时不得出现负值；若外部作弊导致负点，`sanitize` 夹为 0 并打日志。 |
| **超上限** | 任意 capped 属性 >50 的脏存档：加载时**截断为 50**，溢出点数退回 `unspent_points`（并可选记一条修复事件）。 |
| **重置费用** | 第 n 次重置费用 `cost = base * k^(n)` 或线性递增（具体公式 §11）；余额不足禁止执行。 |
| **跨「重入江湖」保留** | 若产品定义重入江湖为周目重置：需明确**修为是否保留**——默认推荐：**保留修为等级与点数分配**（仅重置剧情进度），否则须在 UI 显著告知；若周目清空，则本系统退化为无意义，禁止静默清空。 |
| **离线经验** | 离线结算将按比例生成修为经验，**不超过在线软上限**（如最多折算 24h），且受 `exp_find` 辅助加成影响；上线时批量应用升级与点数。 |
| **未解锁时经验** | 推荐：**不累计**至修为池，避免玩家不知情的「暗箱」；若累计，解锁瞬间一次性转化需防连升爆炸（分段播动画）。 |

## §11 数值公式与战斗对齐

### §11.1 修为经验曲线

升到下一级所需经验（从等级 N 升到 N+1）：

\[
\text{ExpToNext}(N) = \text{BASE} \times (1 + 0.05 \times N)^2
\]

- `BASE`：基础经验常数，用于调「前 20 级约 2 小时一级」等节奏（与击杀频率、修为经验占击杀经验比例联动）。  
- **整数化规则**：实现中建议 `round` 后 `max(MIN, value)`，避免低等级过小；`MIN` 可取 12 量级防除零。  
- **节奏目标（文案级，需配表验证）**：  
  - 前 20 级：约每 **2** 小时 1 级；  
  - 50 级后：约每 **6** 小时 1 级；  
  - 100 级后：约每 **12** 小时 1 级；  
  曲线形态为**多项式增长**，永无硬顶等级。

### §11.2 属性收益（与产品表一致）

| 分类 | 属性 A | 属性 B | 每点收益 | 单属性上限 |
|------|--------|--------|----------|------------|
| 攻击 | 力量 | 攻击力% | 各 **+0.5%**（相对各自池子，见下） | 50 |
| 防御 | 生命值% | 防御力% | 各 **+0.5%** | 50 |
| 技巧 | 暴击率 | 暴击伤害 | **+0.2%** / **+1%** | 50 |
| 辅助 | 碎银获取 | 经验获取 | 各 **+1%** | 无上限 |

**说明**：「力量」与「攻击力%」在战斗公式中须拆入**不同层**：建议力量进「基础攻击/武器系数池」，攻击力%进「全局攻击%乘区」，避免重复定义导致双倍计算。

### §11.3 与 DPS 公式乘区对齐（实现约束）

1. **列出项目现有 DPS 乘区**（示例）：基础攻、攻击%、属性倍率、暴击率、暴击伤害、独立增伤、目标易伤…  
2. 修为提供的每一项必须标注唯一归属：  
   - **攻击力%、力量**：不得同时等于同一乘区的重复输入。  
3. **暴击率**：累加到「暴击率加法池」，并受**硬上限**（如 75%）裁剪时，裁剪顺序为：**先装备与 buff，最后修为**或按项目统一规则——必须在战斗说明文档中单点写明。  
4. **暴击伤害**：加法叠加到暴伤池。  
5. **生命值%、防御力%**：进承伤或有效生命公式，与装备百分比相加而非相乘两次。  
6. **碎银获取 / 经验获取**：仅影响掉落与经验结算，**不进战斗 DPS**。  
7. **修为经验获取加成**：作用于 `gain_experience` 的 `bonus_percent` 链，与辅助中的「经验获取」合并为同一类加成，避免二次叠乘。

### §11.4 重置费用（建议默认）

- `reset_cost = floor(5000 * pow(1.25, reset_count))`，`reset_count` 为已成功重置次数。  
- 存档记录 `reset_count` 与下次预览费用；UI 展示整数碎银。

## §12 测试与验收

### §12.1 经验曲线

- 给定 `BASE`，单元测试 `ExpToNext(0..120)` 单调不减（在 N 足够大前允许平台浮点误差范围内非严格时，用整数数学）。  
- 连升：一次性灌入超大经验，循环减扣与升级次数正确，`paragon_exp` 余数正确。

### §12.2 分配与上限

- 在 `unspent_points = 0` 时确认分配失败。  
- 单属性加到 50 后 +/- 行为；辅助加到 1000 无报错。  
- 战斗系 300 点满后，仅辅助可继续加点。

### §12.3 重置

- 余额充足：全部分配清零，`unspent_points` 恢复总和，`reset_count++`，碎银扣除。  
- 余额不足：状态不变。  
- 重置后立即再分配，战斗数值与 UI 一致。

### §12.4 属性生效

- 对比修改前后：同木桩、同 seed，`damage` 变化与预览误差 < 允许阈值（浮点）。  
- `EventBus` 发射后，HUD 与角色面板同步刷新。

### §12.5 脏数据与迁移

- 超上限、负点、缺失字段存档：经 `sanitize` 后不崩溃且可继续游戏。
