# 离线收益系统 — 开发需求案（§1–§15）

> 面向实现：Godot 4 + GDScript；主逻辑 `scripts/autoload/offline_system.gd`；与 `SaveManager` autoload 持久化对接。  
> 用途：供其他 AI / 程序直接按章节落地离线闭关、模拟结算与「出关所得」展示。

---

## §1 文档目的与范围

**目的**  
定义武侠挂机 RPG 的离线收益闭环：离线期间按规则模拟战斗产出，上线后以「出关所得」弹窗完成仪式感展示与一次性领取，并与存档、保底、分解规则一致。

**范围（含）**  
离线时间截取与上限、战斗节点与效率推算、掉落与分解分流、经验/材料倍率、保底计数器在离线期的累加与持久化、「出关所得」UI 流程、离线设置面板、与 SaveManager 的读写契约、验收与边界。

**范围（不含）**  
具体广告 SDK / 支付 SDK 对接细节（仅保留「2 倍收益」入口与条件占位）；在线战斗每一帧的精确复刻（离线为确定性模拟，见 §6）。

---

## §2 名词与约定

| 名词 | 含义 |
|------|------|
| 闭关 / 离线 | 客户端未运行或不在前台战斗流程，但存档内时间推进并参与结算 |
| 最高通关节点 | 玩家当前已解锁且**已通关**的难度下，进度最高的关卡节点（见 §4） |
| 平均通关时间 | 该节点统计或配置的「单局/单节点」期望耗时，用于换算离线可完成局数 |
| 凡品/精良 | 低品质装备，离线产出后**自动分解为材料**，不进入高光队列 |
| 玄品+ | 玄品及以上品质；进入「出关所得·高光」逐件展示队列 |
| 高光阈值 | **玄品（黄色）及以上**进入高光展示；橙装等更高品质同样进入并强化表现 |
| 出关所得 | 上线后全屏/模态流程：展示离线时长、高光列表、材料与经验汇总、领取 |

**约定**  
- 时间基准：使用存档内可信时间源（见 §10），防篡改策略与现有项目一致。  
- 所有离线产出在**领取前**写入中间态（pending），领取后合并进背包/货币/经验池并清空 pending。

## §3 项目背景与关联系统

- **类型**：武侠挂机 RPG，2D 横版，Godot 4，GDScript。  
- **挂机核心**：离线继续按效率模拟战斗；上线强调「出关」仪式感。  
- **主脚本**：`scripts/autoload/offline_system.gd`（新建或按此路径实现），在 `_ready` 或游戏启动早期向 `SaveManager` 注册/拉取离线字段。  
- **关联**：`loot_system.gd`、`equipment_generator_system.gd`、掉落表与保底字段、分解材料映射表；UI 与 `hud` / 主导航的弹窗层级协调。

---

## §4 离线战斗模拟规则（需求级）

1. **离线上限**：单次结算的离线时长 **≤ 48 小时**；超出部分不计入模拟。  
2. **战斗节点**：始终取 **当前已通关的最高难度节点** 作为模拟战斗源（掉落池、难度系数与该节点在线一致）。  
3. **通关效率**：按该节点 **平均通关时间** 计算；示例：15 秒/波 × 3 波 = 45 秒/节点 → 离线可完成节点数 = `floor(有效离线秒数 / 45)`。若项目以「局」为单位，则改为「每局平均时长」。  
4. **掉落规则**  
   - 凡品/精良：**自动分解为材料**（直接转化，不占用背包格子）。  
   - 玄品+：进入「出关所得·高光」队列，领取时入包或按现有装备入库规则。  
   - **保底机制**：与在线相同规则；离线模拟期间 **保底计数器持续累加** 并持久化（见 §10）。  
5. **经验产出**：相对正常在线战斗结算，效率为 **70%**（见 §11 公式）。  
6. **材料产出**：相对正常在线，效率为 **80%**（分解转化与直接材料之和均受此系数约束，具体实现二选一并在 §11 写死，避免重复缩放）。

---

## §5 「出关所得」展示流程（需求级）

1. 玩家上线（或回到主循环可交互态）。  
2. **全屏**：「闭关修炼已毕」+ **离线时长**（格式化：X 天 X 小时 / 或 X 小时 X 分）。  
3. **逐条展示高光**：玄品+；**每件约 1.5 秒**，含 **光柱动画**（橙装及以上可加强粒子/颜色）。  
4. **材料总览**：碎银、锻材、精铁等汇总数量（与分解转化一致）。  
5. **经验获取总量**。  
6. **「领取全部」** 按钮：一次性提交 pending → 正式加到玩家数据。  
7. **可选**：「2 倍收益」——观看广告或消耗元宝（占位：需二次确认、失败回滚、仅对未领取 pending 生效或按产品规则在 §11 锁定）。

## §6 L1–L5 体验设计（本需求新写）

- **L1 Fantasy**：「闭关数日，出关所得堆积如山」——玩家感知角色在闭关修炼，上线即见成果堆叠。  
- **L1 Emotion**：主情绪为 **回归惊喜**；次情绪为 **积累满足**（数字与高光强化「没白离线」）。  
- **L1 Payoff**：打开 APP →「闭关修炼已毕」→ 高光逐件 → 橙装光柱 → 材料「瀑布」式总览 →「领取全部」。  
- **L2**  
  - **秒**：回归瞬间仪式动画（全屏标题 + 时长 + 轻震/音效）。  
  - **分**：出关所得展示总时长目标 **约 1–3 分钟**（随高光件数浮动，可跳过单件但默认不跳过）。  
  - **时**：离线 **2–4 小时** 即有可感知材料与经验增量（与 §11 公式对齐）。  
  - **日**：每日至少一次离线回归，能完整走一遍「出关」主流程（可配合日常任务，非本文件必选）。  
- **L5**：**出关仪式** = 全屏动画 + 高光逐件 + 收益统计 + 明确 CTA（领取）。

## §8 数据结构与 SaveManager 字段（建议）

下列字段名可调整，但须在文档与代码中 **一对一映射**：

- `offline_last_timestamp`：上次可靠下线时间戳。  
- `offline_pending_*`：未领取收益包（装备 id 列表、材料字典、经验值、分解明细、模拟用的随机种子或版本号）。  
- `pity_counters`：与在线共用或显式同步；离线模拟 **只加不减** 直至掉落触发重置。  
- `offline_settings`：分解阈值、是否允许「稍后领取」等。  
- `offline_sim_version`：用于跨版本迁移（见 §10）。

**领取原子性**：领取接口在一个事务内：校验 pending → 加背包/货币/经验 → 清空 pending → 写存档；失败则不打断旧 pending。

## §9 完整操作流

1. **离线**：玩家关闭应用或断线；运行时 `SaveManager` 写入 `offline_last_timestamp` 及必要战斗快照（最高通关节点 id、倍率、保底计数）。  
2. **模拟计算**：下次启动，`offline_system` 读取当前可信时间与 `offline_last_timestamp`，计算 **有效离线秒数**（截断 48h），换算节点局数，循环调用与在线共享的掉落/经验/材料接口（带 offline 标记）。  
3. **上线**：若 `pending` 非空或本次刚生成 pending，进入 **阻塞式出关流程**（或主菜单优先队列）。  
4. **弹窗**：展示 §7.1 全屏页。  
5. **展示**：按高光队列逐件播放；材料与经验区可在高光结束后一次性填满或逐步填入（需与动画一致）。  
6. **领取**：点击「领取全部」执行 §8 原子领取。  
7. **进入游戏**：关闭弹窗，恢复 HUD/操作；若有二级确认（2 倍收益），在领取前或领取后按产品规则分支。

## §10 工程约束：时间、版本、保底、中断

| 类别 | 方案 |
|------|------|
| **离线上限** | 有效离线时间 `min(Δt, 48h)`；UI 与模拟必须使用同一截断结果。 |
| **跨版本存档兼容** | `offline_sim_version` 与迁移表：新版本可增加字段，旧 pending 按默认材料化或按迁移函数折算；禁止静默吞掉高光装备。 |
| **保底计数器持久化** | 离线模拟循环内每次「虚拟战斗」与在线一致更新计数；存档写入频率可批处理，但领取前必须已落盘最新计数。 |
| **中断恢复** | 应用崩溃或杀进程：已写入的 `pending` 不丢失；下次启动继续展示；「领取全部」前不重复结算同一段离线时间（用 `offline_last_timestamp` 在领取后更新为「已结算锚点」）。 |

## §11 收益公式与 L2 节奏对齐

**符号**  
- `T_eff = min(t_now - t_last, 48h)`  
- `T_node`：当前最高通关节点平均通关时间（秒）  
- `N = floor(T_eff / T_node)`：离线完成的节点次数  

**经验（对齐 L2「2–4h 有感」）**  
- `Exp_offline = 70% × Σ(i=1..N) Exp_online(node)`  
- 若在线按波次给经验，则对波次求和再 ×0.7，与 `N` 定义一致。

**材料**  
- `Mat_offline = 80% × ( Mat_drop + Mat_decompose )`  
- **实现二选一写死**：要么只对掉落包缩放 0.8，要么只对分解产出缩放 0.8，**禁止双端叠乘**。  

**高光与分解**  
- 每件装备先判定品质；凡品/精良 → 直接映射材料表；玄品+ → 进高光队列并在领取时入库。  

**节奏校验**  
- 用 §6 L2「时」目标反推：在典型中期节点下，离线 2h、4h 的材料与经验增量须有 **可展示的非零结果**；若 `N=0`，仍允许有保底进度增长但 UI 文案需温和（避免「空窗羞辱」）。

## §12 验收标准

| 维度 | 要求 |
|------|------|
| **离线计算正确性** | 同一输入（时间戳、种子、节点）离线两次结算结果一致；与在线同节点单次期望偏差在文档允许倍率（0.7/0.8）内。 |
| **出关展示完整性** | 高光件数与 pending 列表一致；材料总和与分解日志一致；经验总值与公式一致。 |
| **边界条件** | 见下表 |

**边界条件（每类 1–2 句具体方案）**

| 边界 | 方案 |
|------|------|
| 无通关节点 / 新手 | 不进行战斗模拟或仅给保底时间进度；UI 展示「尚未闭关」并跳过出关页。 |
| `T_eff < T_node` | `N=0`：不产出局内掉落，仍可更新在线外系统（若有）；经验材料可为 0。 |
| pending 已满背包 | 领取前校验格子：不足则提示整理背包或只领取非装备部分（需产品定夺，推荐阻塞领取装备）。 |
| 改系统时间 | 与项目全局反作弊策略一致：负向 Δt 视为 0；异常大正向可截断或服务器校准（若仅有本地则截断 48h）。 |
| 跨天/活动倍率 | 离线是否吃活动：默认 **不吃** 或按 `offline_settings` 只读展示；若吃，须在 SaveManager 记下活动 id 防重。 |
| 重复进入弹窗 | 同一段 pending 只展示一次直至领取或明确「稍后」；防重复结算见 §10。 |

## §13 非功能需求

- **性能**：万级分解次数在批处理内完成，避免单帧卡顿；必要时进度条「正在整理出关行囊」。  
- **可测试性**：暴露 `simulate_offline_for_debug(seconds)`（仅 Debug 编译）。  
- **本地化**：时长与数字格式走统一工具。  
- **无障碍**：关键按钮大触控区；光柱动画可关「减弱动态效果」。
