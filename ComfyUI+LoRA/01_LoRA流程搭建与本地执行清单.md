# LoRA 流程搭建与本地执行清单

## 1. 目标

本文档用于明确本地 `ComfyUI + LoRA` 的推荐工作流，避免后续把：

- 训练
- 推理
- 测试
- 验收

全部混在一起，导致工作区越来越乱。

## 2. 推荐工具链

基于当前机器上已有的使用痕迹，推荐采用如下分工：

### 2.1 ComfyUI

用于：

- 文生图/图生图
- LoRA 挂载测试
- 批量出图
- 验收对比图
- 工作流保存

不建议把 ComfyUI 作为唯一训练入口。

### 2.2 训练工具

建议二选一，不要同时维护两套训练参数。

优先推荐：

- `kohya_ss / sd-scripts`

可选：

- `OneTrainer`

## 3. 推荐实际工作流

建议采用下面这条闭环：

1. 在 `00_reference` 收集风格参考和目标资源样式
2. 在 `01_dataset` 收集原始图
3. 在 `02_processed` 做尺寸、命名、caption 清洗
4. 用 `03_configs` 中的训练配置启动 LoRA 训练
5. 将权重输出到 `04_output/loras`
6. 在 `05_workflows` 保存对应 ComfyUI 测试工作流
7. 在 `06_test_prompts` 用固定测试 prompt 验证风格与稳定性

## 4. 推荐 LoRA 拆分

对于 `桌面挂机`，不建议做一套大杂烩 LoRA。

建议拆分如下：

### 4.1 style_lora

学习内容：

- 灾后黑铁废土
- 工业遗迹
- 熔核科技残骸
- 深灰、锈红、炉橙、冷蓝的整体风格

用途：

- 角色
- 背景
- 宣传图
- 整体产品风格统一

### 4.2 hero_lora

学习内容：

- 主角形象
- 主角装备轮廓
- 主角兵装风格

用途：

- 角色立绘
- 主角宣发图
- 主角战斗姿态参考

### 4.3 enemies_lora

学习内容：

- 异化兽群
- 守卫残军
- 熔灾信徒
- Boss 外观

用途：

- 怪物概念图
- Boss 设计稿
- 精英与敌人族群扩展

### 4.4 items_icons_lora

学习内容：

- 技能图标
- 装备图标
- 资源图标
- 稀有边框
- 结果卡图形元素

用途：

- UI 图标
- 结果卡
- 品质边框

### 4.5 backgrounds_lora

学习内容：

- 废墟边境
- 焦灼矿场
- 后续章节环境主题

用途：

- 章节背景
- 宣传图背景
- 关卡氛围参考

## 5. 当前最建议的起步顺序

如果现在就开始搭流程，推荐顺序如下：

1. 先做 `style_lora`
2. 再做 `items_icons_lora`
3. 再做 `backgrounds_lora`
4. 最后再做 `hero_lora` 与 `enemies_lora`

原因：

- 风格 LoRA 最先解决“世界观统一”
- 图标/UI LoRA 最先服务现阶段产品实现
- 背景 LoRA 能快速提升截图质量
- 角色与敌人 LoRA 最复杂，适合在前面流程稳定后再训

## 6. 本地执行前检查清单

在真正训练之前，先确认：

- 本地 `ComfyUI` 是否可启动
- 是否已有固定底模
- 是否已选定唯一训练工具
- 是否确定第一套 LoRA 的目标
- 是否已有至少一版整理过的数据集
- 是否定义好测试 prompt

只要上面任意一项没准备好，就不建议直接开训。

## 7. 首批推荐目录使用方式

### 7.1 `00_reference`

放：

- 色板图
- 风格参考图
- 角色参考图
- 章节环境参考图
- UI 风格参考图

### 7.2 `01_dataset`

放原始训练图。

要求：

- 同一子类单独存放
- 不混不同目标
- 暂时不要把处理后副本和原图混在一起

### 7.3 `02_processed`

放：

- 重命名后的图
- 统一尺寸后的图
- 配套 caption

### 7.4 `03_configs`

放：

- 每一套 LoRA 的训练参数配置
- 实验版本说明
- 学习率、epoch、rank 等记录

### 7.5 `05_workflows`

建议至少维护 3 套工作流：

- 单图测试工作流
- 批量角色测试工作流
- 批量 UI 图标测试工作流

## 8. 训练后验收方式

训练完一套 LoRA 后，不要只看“能不能出图”，而要看：

- 风格是否统一
- 是否过拟合
- 是否把角色、背景、图标学串了
- 是否符合 `桌面挂机` 当前题材方向
- 是否能直接服务项目当前资源缺口

## 9. 当前结论

本地最推荐的路线不是“ComfyUI 包打一切”，而是：

- `ComfyUI` 负责推理、工作流、验收
- `kohya_ss` 或 `OneTrainer` 负责训练
- 本目录负责管理数据、配置、输出与测试规范

这样最稳，也最适合长期维护。
